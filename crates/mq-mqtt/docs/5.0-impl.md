## 2. Detailed Gap Analysis

### GAP-1: Will Delay Interval (MQTT 5.0 SS 3.1.3.2.2)

**Spec requirement:** When a client disconnects ungracefully and the Will Flag is set, the server MUST delay publishing the Will Message by `will_delay_interval` seconds (property 0x18). If the client reconnects to the same session before the delay expires, the Will Message MUST NOT be published. If the session expires before the delay, the Will Message is published at session expiry.

**Current state:** The `will_delay_interval` property is parsed from CONNECT Will Properties and stored in the session, but the server publishes the Will Message immediately on disconnect — the delay is ignored.

**Impact:** High — violates a MUST-level requirement. Clients relying on grace-period reconnect (common in mobile/IoT) will see spurious will messages.

**Fix:**
- On ungraceful disconnect, if `will_delay_interval > 0`, spawn a `tokio::time::sleep` task keyed by `(client_id, session_id)`.
- If the client reconnects to the same session before expiry, cancel the pending task.
- If the session expires first (session_expiry_interval < will_delay_interval), publish at session expiry.
- Store the `JoinHandle` or `AbortHandle` in `SessionState` so it can be cancelled on reconnect.

### GAP-2: Message Expiry Interval Enforcement (MQTT 5.0 SS 3.3.2.3.3)

**Spec requirement:** The server MUST delete a message whose `message_expiry_interval` (property 0x02) has elapsed before delivery. When forwarding, the server MUST set the interval to the received value minus the time the message has been waiting (i.e. subtract elapsed time).

**Current state:** The property is parsed and stored as TTL metadata in `FlatMessage`, but:
1. No reaper/eviction mechanism drops expired messages from queues.
2. The elapsed-time subtraction is not applied when forwarding to subscribers.

**Impact:** High — expired messages are delivered, violating a MUST requirement.

**Fix:**
- Before delivering any queued message, check `publish_time + message_expiry_interval < now` and skip if expired.
- When forwarding, rewrite property 0x02 to `original_interval - elapsed_seconds` (clamped to 0).
- Optionally add a background sweep for long-lived queues to reclaim storage.

---

### GAP-3: Retain Handling Subscription Options (MQTT 5.0 SS 3.3.1-9/10/11)

**Spec requirement:** The Retain Handling field (bits 4-5 of subscription options byte) controls retained message delivery on subscribe:
- **0**: Send retained messages at subscribe time.
- **1**: Send retained messages only if the subscription did not already exist.
- **2**: Do not send retained messages at subscribe time.

**Current state:** All three values are parsed from SUBSCRIBE, but the server always delivers retained messages on subscribe (mode 0 behavior), regardless of the requested mode.

**Impact:** Medium — affects clients that explicitly opt out of retained message floods on resubscribe (e.g. dashboard reconnects).

**Fix:**
- In the subscribe handler, check `retain_handling`:
  - `0` → query and deliver retained messages (current behavior).
  - `1` → query and deliver only if this is a new subscription (not already in session state).
  - `2` → skip retained message delivery entirely.

### GAP-4: Retain As Published (MQTT 5.0 SS 3.3.1-12/13)

**Spec requirement:**
- If the subscription's Retain As Published flag = 0, the server MUST clear the RETAIN flag (set to 0) when forwarding application messages to that subscriber.
- If the flag = 1, the server MUST preserve the original RETAIN flag from the received PUBLISH.

**Current state:** The option is parsed but not consulted during message forwarding. The RETAIN flag is passed through unchanged in all cases.

**Impact:** Medium — clients that expect RETAIN=0 on forwarded messages will see incorrect flag values.

**Fix:**
- In the delivery path (`session.rs` publish-to-subscriber logic), read the subscription's `retain_as_published` flag.
- If `false`, mask off the RETAIN bit in the outgoing PUBLISH fixed header.

---

### GAP-5: Maximum QoS Enforcement (MQTT 5.0 SS 3.2.2-11)

**Spec requirement:** If the server advertises `maximum_qos` (property 0x24) in CONNACK, the client MUST NOT send PUBLISH packets with QoS exceeding that value. If the client does, the server MUST treat it as a Protocol Error — send DISCONNECT with reason code 0x9B (QoS not supported) and close the connection.

Additionally, the server MUST downgrade subscription grants: `granted_qos = min(requested_qos, maximum_qos)`.

**Current state:** The server advertises `maximum_qos` in CONNACK properties but does not:
1. Reject inbound PUBLISH packets that exceed it.
2. Downgrade granted QoS in SUBACK.

**Impact:** Medium — a misbehaving client can force QoS 2 flows even when the server advertises QoS 1 max.

**Fix:**
- In the PUBLISH handler, if `publish.qos > server_maximum_qos`, send DISCONNECT 0x9B.
- In the SUBSCRIBE handler, clamp each topic filter's QoS: `granted = min(requested, server_max_qos)`.

### GAP-6: No Local Option (MQTT 5.0 SS 3.8.3-3/4)

**Spec requirement:**
- If No Local = 1, the server MUST NOT forward a message to a connection that published it (matched by ClientID).
- No Local MUST NOT be set to 1 on Shared Subscriptions (Protocol Error).

**Current state:** The `no_local` flag is parsed from subscription options but never checked during message delivery. Shared subscription validation for No Local is also missing.

**Impact:** Medium — clients subscribing to their own publish topic (common in request/response patterns) will receive their own messages.

**Fix:**
- In the delivery path, compare the publisher's `client_id` against the subscriber's `client_id`; skip if `no_local && publisher_client_id == subscriber_client_id`.
- In SUBSCRIBE validation, reject `no_local = 1` on `$share/` topic filters with reason code 0xA2.

---

### GAP-7: Shared Subscriptions (MQTT 5.0 SS 4.8.2)

**Spec requirement:**
- Topic filters starting with `$share/{ShareName}/{filter}` denote shared subscriptions.
- Messages matching a shared subscription are delivered to only ONE subscriber in the group (load balanced).
- Retained messages MUST NOT be sent when a shared subscription is established.
- No Local MUST NOT be 1 for shared subscriptions.

**Current state:** The `$share/` prefix is parsed and the share group name extracted. Queue naming supports group semantics. However:
1. Load-balanced delivery to exactly one group member is unverified.
2. Retained message suppression for shared subscriptions is not implemented.
3. No Local validation for shared subscriptions is missing (see GAP-6).

**Impact:** Medium — shared subscriptions are a key MQTT 5.0 feature for scalable consumer groups.

**Fix:**
- Verify that the bisque-mq queue binding delivers to exactly one consumer per group.
- Suppress retained message delivery when the subscription is shared.
- Add No Local validation (see GAP-6).
- Add integration tests for shared subscription round-trip.

---

### GAP-8: Topic-Level Authorization (MQTT 5.0 SS 5.4.2)

**Spec requirement:** Servers SHOULD implement topic-level access control. If a client is not authorized to publish or subscribe to a topic, the server should return appropriate reason codes (0x87 Not Authorized).

**Current state:** `AuthProvider::authorize_topic()` exists as a trait method with a default implementation that always returns `true`. The method is never called from the server's PUBLISH or SUBSCRIBE handlers.

**Impact:** Medium — any authenticated client can publish/subscribe to any topic.

**Fix:**
- Call `auth_provider.authorize_topic(client_id, topic, operation)` in both:
  - PUBLISH handler → return PUBACK/PUBREC with 0x87 on deny.
  - SUBSCRIBE handler → return per-filter 0x87 in SUBACK on deny.
- Document the `AuthProvider` trait so users can plug in ACL implementations.

---

### GAP-9: Will Properties Preservation (MQTT 5.0 SS 3.1.3.2)

**Spec requirement:** When publishing a Will Message, the server MUST include the original Will Properties: Payload Format Indicator, Message Expiry Interval, Content Type, Response Topic, Correlation Data, and User Properties. User Property ordering MUST be maintained.

**Current state:** Will properties are parsed from CONNECT and stored in the `WillMessage` struct. However, when the Will Message is converted to a `FlatMessage` for publication, not all properties may survive the conversion (particularly User Properties and Content Type).

**Impact:** Low-Medium — affects clients that rely on will message metadata (e.g. last-will status with content type).

**Fix:**
- Audit the `WillMessage → FlatMessage` conversion path.
- Ensure all 6 will properties are mapped to the corresponding FlatMessage fields.
- Add a round-trip test: CONNECT with will properties → disconnect → verify published will includes all properties.

---

### GAP-10: Server Redirection (MQTT 5.0 SS 4.13)

**Spec requirement:** The server MAY send CONNACK or DISCONNECT with reason codes 0x9C (Use another server) or 0x9D (Server moved), including a Server Reference property (0x1C) pointing to an alternative server.

**Current state:** The `server_reference` property and reason codes are defined and parseable, but there is no server-side mechanism to trigger redirection (e.g. during maintenance or load balancing).

**Impact:** Low — this is a MAY-level feature. No spec violation if absent.

**Fix (optional):**
- Add a configuration option or runtime API to put the server in "redirect mode".
- When active, respond to new CONNECT packets with CONNACK 0x9C/0x9D + Server Reference.
- Useful for graceful shutdown / rolling deployments.

---

### GAP-11: Message Rate & Quota Limiting (MQTT 5.0 SS informative)

**Spec requirement:** The server MAY disconnect clients that exceed message rate limits (0x96) or quotas (0x97). These are server-policy features, not strict protocol requirements.

**Current state:** Reason codes 0x96 and 0x97 are defined but no rate-limiting or quota enforcement exists.

**Impact:** Low — MAY-level. No spec violation. Important for production hardening.

**Fix (optional):**
- Add configurable per-client rate limits (messages/sec) using a token bucket or sliding window.
- On exceed, send DISCONNECT with 0x96.
- Add configurable per-client storage quotas; on exceed, reject PUBLISH with PUBACK 0x97.

---

### GAP-12: Session Expiry Interval Override on DISCONNECT (MQTT 5.0 SS 3.14.2.2.2)

**Spec requirement:** A client MAY include `session_expiry_interval` in DISCONNECT to override the value from CONNECT. However, if CONNECT set the interval to 0, a non-zero value in DISCONNECT is a Protocol Error.

**Current state:** Needs verification — the DISCONNECT handler may not read/apply the session_expiry_interval property from the DISCONNECT packet, and the zero-to-nonzero validation may be missing.

**Impact:** Low-Medium — affects clients that want to extend or shorten session lifetime at disconnect time.

**Fix:**
- In the DISCONNECT handler, read `session_expiry_interval` from properties.
- If present and CONNECT's value was 0, send DISCONNECT 0x82 (Protocol Error).
- Otherwise, update the stored session expiry with the new value.

## 3. Implementation Plan

### Phase 1 — Critical MUST-Level Fixes

These gaps violate MUST-level requirements and should be fixed first.

| # | Gap | Effort | Files |
|---|-----|--------|-------|
| 1 | Will Delay Interval | Medium | `session.rs`, `server.rs` |
| 2 | Message Expiry Interval enforcement | Medium | `session.rs`, `server.rs` |
| 5 | Maximum QoS enforcement | Small | `session.rs`, `server.rs` |

**Phase 1 tasks:**

1. **Will Delay Interval**
   - Add `will_delay_handle: Option<AbortHandle>` to session state.
   - On ungraceful disconnect with `will_delay_interval > 0`, spawn delayed task.
   - On session reconnect, abort pending will task.
   - On session expiry, publish will immediately if delay still pending.
   - Test: connect with will_delay=5s → disconnect → reconnect within 5s → verify no will published.

2. **Message Expiry Interval**
   - Before delivering any message, check `created_at + message_expiry_interval < now()`.
   - When forwarding, rewrite property 0x02: `new_interval = original - elapsed`.
   - If `new_interval <= 0`, skip delivery.
   - Test: publish with expiry=1s → wait 2s → subscribe → verify message not delivered.

3. **Maximum QoS Enforcement**
   - In PUBLISH handler: if `qos > server_max_qos`, send DISCONNECT 0x9B.
   - In SUBSCRIBE handler: `granted_qos = min(requested, server_max_qos)`.
   - Test: set server max_qos=1 → client publishes QoS 2 → verify disconnect.

### Phase 2 — Subscription Option Enforcement

These fix MUST-level subscription behaviors.

| # | Gap | Effort | Files |
|---|-----|--------|-------|
| 3 | Retain Handling options | Small | `session.rs`, `server.rs` |
| 4 | Retain As Published | Small | `session.rs` |
| 6 | No Local option | Small | `session.rs` |
| 12 | Session Expiry on DISCONNECT | Small | `session.rs` |

**Phase 2 tasks:**

4. **Retain Handling**
   - In subscribe path, branch on `retain_handling`:
     - `0` → send retained (current).
     - `1` → send only if subscription is new (not in existing session subscriptions).
     - `2` → skip retained entirely.

5. **Retain As Published**
   - In delivery path, if subscription `retain_as_published == false`, clear RETAIN bit.

6. **No Local**
   - In delivery path, skip if `no_local && publisher_client_id == subscriber_client_id`.
   - In SUBSCRIBE validation, reject `no_local = 1` on `$share/` filters.

7. **Session Expiry Override**
   - Read `session_expiry_interval` from DISCONNECT properties.
   - Validate zero-to-nonzero transition → Protocol Error.
   - Update stored session expiry.

### Phase 3 — Shared Subscriptions & Authorization

| # | Gap | Effort | Files |
|---|-----|--------|-------|
| 7 | Shared Subscriptions | Medium | `session.rs`, `server.rs` |
| 8 | Topic-Level Authorization | Medium | `auth.rs`, `session.rs`, `server.rs` |
| 9 | Will Properties Preservation | Small | `session.rs`, `codec.rs` |

**Phase 3 tasks:**

8. **Shared Subscriptions**
   - Verify bisque-mq queue semantics deliver to exactly one consumer per share group.
   - Suppress retained messages for shared subscriptions.
   - Add integration tests for `$share/group/topic` with multiple subscribers.

9. **Topic Authorization**
   - Call `auth_provider.authorize_topic()` in PUBLISH and SUBSCRIBE handlers.
   - Return 0x87 on deny.
   - Document the trait for custom ACL implementations.

10. **Will Properties**
    - Audit `WillMessage → FlatMessage` conversion.
    - Map all 6 will properties (payload_format_indicator, message_expiry_interval, content_type, response_topic, correlation_data, user_properties).
    - Add round-trip test.

### Phase 4 — Production Hardening (Optional / MAY-level)

| # | Gap | Effort | Files |
|---|-----|--------|-------|
| 10 | Server Redirection | Small | `server.rs` |
| 11 | Rate & Quota Limiting | Medium | `session.rs`, `server.rs` |

**Phase 4 tasks (optional):**

11. **Server Redirection**
    - Add `redirect_mode` config with target server reference.
    - When active, respond to CONNECT with CONNACK 0x9C + server_reference.

12. **Rate Limiting**
    - Token-bucket per client for publish rate.
    - DISCONNECT 0x96 on exceed.
    - Per-client queue depth quota; PUBACK 0x97 on exceed.

## 4. What's Already Solid

The following areas are fully implemented and spec-compliant:

- **All 15 control packet types** — CONNECT, CONNACK, PUBLISH, PUBACK, PUBREC, PUBREL, PUBCOMP, SUBSCRIBE, SUBACK, UNSUBSCRIBE, UNSUBACK, PINGREQ, PINGRESP, DISCONNECT, AUTH.
- **All 27 MQTT 5.0 properties** — parsed, written, and validated (including multi-occurrence User Properties, Subscription Identifiers).
- **QoS 0/1/2 complete state machines** — packet ID tracking, in-flight counters, DUP flag on session resume, retransmit only on reconnect with existing session.
- **Session state management** — Clean Start, session_expiry_interval with periodic 60s sweep, DashMap-backed store, session present flag.
- **Enhanced authentication** — AUTH packet, multi-step challenge/response, re-authentication (reason code 0x19), pluggable AuthProvider trait.
- **Flow control** — Receive Maximum enforced in both directions, DISCONNECT 0x93 on exceed, Maximum Packet Size checked before send.
- **Topic aliases** — bidirectional alias maps, validation (reject 0, reject > maximum), per-connection lifecycle.
- **Keep alive** — Server Keep Alive override, 1.5x timeout enforcement.
- **All 40+ reason codes** — defined and contextually used across CONNACK, PUBACK, PUBREC, PUBREL, PUBCOMP, SUBACK, UNSUBACK, DISCONNECT.
- **UTF-8 validation** — surrogate pairs, control characters, null bytes, non-characters.
- **Topic filter matching** — multi-level `#`, single-level `+`, `$`-prefixed topic exclusion from wildcard subscriptions.
- **Request Problem Information** — strips reason strings and user properties from non-CONNACK/DISCONNECT packets when flag = 0.
- **MQTT 3.1.1 backward compatibility** — protocol version detection, clean_session mapping, v3.1.1 subscription options validation.
